# OpenAI Prompting Best Practices (Snapshot: 2026-02-25)

このメモは `prompt-eval-improver` Skill 用の参照ノートです。OpenAI 系（ChatGPT / OpenAI API / Codex）向けを主対象にしています。日付依存のため、ユーザーが「最新」を求める場合は再確認してください。

## 使い方

- OpenAI 系モデル向けにプロンプト改善する場合に読む
- モデル固有の推奨事項（例: GPT-5.2）と、一般原則を分けて適用する
- ルールをそのまま転写せず、対象タスクに合わせて最小限を採用する

## 要点（一般原則）

1. まず評価基準を決める（デフォルトは 10項目×10点でもよい）

- 改善前に「何を良くしたいか」を固定する
- 品質評価は単一スコアだけでなく、観点別に分解する
- 10項目×10点を使う場合でも、合計点だけでなく未達観点を記録する

2. 先にシンプルなプロンプトで基準線を作る

- 初手から過剰に複雑化しない
- 最小構成で失敗パターンを確認してから制約を追加する

3. 入出力契約を明確化する

- 入力前提、出力形式、禁止事項、不明時の扱いを明示する
- 形式要件が強い場合は、例やスキーマを短く提示する

4. 追加指示は優先順位を明確にする

- 必須要件と改善提案を混在させない
- 矛盾時の優先順を明示すると逸脱が減る

5. 検証しやすい設計にする

- 比較前提（改善前/改善後）を残す
- 失敗例・境界例をテストに含める

## 要点（GPT-5.2 / 最新ガイド寄り）

1. モデル固有の挙動を前提にしすぎない

- 旧バージョン前提の固定文言を流用しない
- モデル更新時に崩れやすい箇所（自己採点閾値、冗長な強制手順）を疎結合にする

2. 「考え方の強制」より「タスク定義と制約」を優先する

- 推論過程の詳細出力を強制するより、必要な出力契約を明確化する
- 内部推論に依存した品質担保設計を避ける

3. Prompt Optimizer は下書き生成として使い、人手でレビューする

- 高品質な制約や観点を抽出する補助として使う
- 生成結果をそのまま採用せず、過剰制約や重複を削る

4. 評価の過学習を避ける

- 固定サンプルだけで最適化しすぎない
- 未見ケースや反例で再評価する

5. モデル別チューニングを分離する

- 本体プロンプト（移植しやすい部分）と、実行パラメータやモデル依存推奨を分ける
- 例: 推論量・verbosity は外部設定欄として持たせる

## 旧プロンプトから活かせる点（参考）

- 評価観点を明示している点
- 不明点の停止条件を定義している点
- 改善前/改善後比較と差分説明を求めている点
- 運用視点（入力テンプレ、導入手順）まで見ている点

## 旧プロンプトの見直しポイント（参考）

1. 固定フォーマットの拘束が強すぎる

- 常に `1〜11` の順序固定にすると、軽量タスクでも重くなる
- タスクに応じた簡易版 / 詳細版の切替を許容する

2. 自己採点 95 点閾値は形骸化しやすい

- 高い固定閾値は、説明を膨らませる動機になりやすい
- 代わりに「未達観点」「追加情報」「次の一手」を出させる方が実務向き

3. 自己訂正ループ回数の固定は目的依存

- ループ回数より、どの失敗を修正したかを明示させる方が有効

4. ガイド参照の前提が曖昧

- 「準拠する」とだけ書くと、参照不能時の品質差が大きい
- 出典名・日時・適用範囲を明示する

## 改善設計に反映するルール

- 必須要件 / 任意改善 / 実行パラメータを分離する
- スコアは比較の補助として使い、合計点だけでなく観点別の未達理由も必ず示す
- 追加質問はブロッカー時のみ行い、通常は仮定明示で前進する
- 出力は「コピペ可能な改善後プロンプト」と「変更理由の追跡性」を最優先にする
