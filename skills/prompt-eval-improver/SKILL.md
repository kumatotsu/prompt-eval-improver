# Prompt Eval Improver

ユーザーが提示したプロンプトを、実務で使える水準まで分析・評価・改善する。

## 目的

- 改善後プロンプトをコピペ可能な形で提示する
- 改善理由を追跡可能な評価レポートを提示する
- 実行環境に応じた改善形（ChatGPT / API / Codex）まで落とし込む
- 必要に応じて、追加質問・評価観点・テストケースを提示する

## 進め方

1. レビュー対象と実行環境を確認する
2. 軽量レビューか詳細レビューかを決める
3. 現行プロンプトを分解して問題を特定する
4. 実行環境に合う改善案を作る
5. 差分・理由・リスク・運用上の注意を整理する
6. 評価方法（確認手順 / テスト観点 / eval 案）を提示する
7. ブロッカーがある場合のみ追加質問する

## 入力確認（最初に見る）

最低限、以下を確認する。未提示なら「合理的な仮定」を明示して前進する。

- 対象プロンプト（全文）
- 想定モデル / 実行環境（例: ChatGPT, OpenAI API, Codex）
- タスク種別（生成 / 抽出 / 変換 / 分類 / エージェント制御 / 評価）
- 成功条件（何ができれば良いか）
- 制約（形式、長さ、禁止事項、安全、機密、レイテンシ、コスト）

## レビューモード

デフォルトは軽量レビューとし、必要な場合だけ重くする。

- 軽量レビュー:
  - 問題点を優先度順に 3〜5 個へ絞る
  - 採点表は省略し、改善後プロンプトと検証観点を優先する
- 詳細レビュー:
  - ユーザーが採点を明示的に求めた場合
  - 比較レビューや高リスク用途で、観点別の追跡が必要な場合
  - このときのみ `10項目×10点` を既定で使う

## 追加質問のルール（最小限）

以下に該当する場合のみ質問する。まとめて 1〜3 問までに抑える。

1. 成果物の形式や採点基準が確定できない
2. 法務 / 安全 / 機密の制約が不明
3. 目的が曖昧で改善方針が複数に割れる
4. モデル依存の最適化が必要なのに対象モデルが不明

該当しない場合は停止せず、仮定を明示して続行する。

## 実行環境別の分岐

実行環境ごとに、改善案の形を変える。

- ChatGPT:
  - 会話内でそのまま使う 1 本のプロンプトとして改善する
  - API 固有の設定名や内部実装詳細は本文へ混ぜすぎない
- OpenAI API:
  - system/developer/user の責務分離か、prompt object での変数化を優先する
  - `reasoning_effort` やツール設定など、実行時パラメータは本文から分離して提案する
  - 必要なら prompt object 案と最小 eval 案を添える
- Codex / agentic:
  - tool policy、verification loop、stop condition、parallel / serial 実行方針を明示する
  - 長時間フローでは `commentary` と `final_answer` の分離を意識する
  - tool search や computer use が関係する場合は、その前提条件も書く

## 分析観点（標準）

以下を基準に分析する。軽量レビューでは重要観点に絞り、詳細レビューでは 10項目×10点（各項目 0〜10 点）で評価する。

1. 目的適合性（目的と出力が一致しているか）
2. 入出力契約（入力前提 / 出力形式 / 例外時の扱い）
3. 制約の明確さ（禁止事項、範囲、優先順位）
4. 曖昧性耐性（解釈のぶれへの強さ）
5. 再現性（同種入力で安定しやすいか）
6. 実行環境適合性（ChatGPT / API / Codex に噛み合っているか）
7. ツール戦略（必要時のみ。tool policy / 並列化 / 検証ループが適切か）
8. 検証可能性（良し悪しを判定できるか。eval や grader を作りやすいか）
9. 実運用性（保守性、流用性、コスト/速度調整のしやすさ）
10. 失敗時設計（不明時の応答、確認質問、フォールバック）

## 改善方針（GPT-5.4 時代のベストプラクティス寄り）

- 先に目的と成功条件を固定し、その後に文面を最適化する
- 役割定義だけに依存せず、output contract と completeness contract を明示する
- 入力境界を区切る（デリミタ、セクション化、変数化）
- 出力形式を具体化する（箇条書き / JSON / 表 / コードブロック等）
- 必要な例だけ示す（少数の高品質例、不要な冗長例は避ける）
- モデル依存の制御（`reasoning_effort`、ツール選択、phase など）は本文から分離する
- エージェント系では verification loop、tool persistence rules、stop condition を定義する
- Prompt object や eval へ載せ替えやすい形へ寄せる
- 固定スコア閾値や自己再出力ループは、無限ループや自己採点の形骸化を避ける設計にする
- 追加提案は「必須要件」と分離して提示する
- 高リスク用途では評価計画（テストケース / 失敗例）を必ず付ける

## 出力要件（基本）

ユーザー要望がない限り、以下の順で簡潔に出す。

1. 前提と仮定
2. 現行プロンプトの問題点（優先度順）
3. 改善後プロンプト（コピペ可能）
4. 実行環境に応じた補足（ChatGPT / API / Codex）
5. 差分要約（何を、なぜ変えたか）
6. 確認用テスト観点 / 失敗ケース例 / eval 案
7. 評価レポート（詳細レビュー時のみ）
8. 追加質問（必要時のみ）

## 改善後プロンプトの作り方

改善後プロンプトには、必要に応じて以下の要素を含める。

- ミッション / 目的
- 対象タスクの定義
- 入力仕様（前提、欠損時の扱い）
- 出力仕様（形式、順序、粒度）
- 制約（禁止事項、優先順位、安全）
- ツール利用方針（必要な場合のみ）
- 判断不能時の扱い（不明 / 仮定 / 追加質問）
- 評価や自己確認の方法（必要な場合のみ）

以下は原則避ける。

- 実質的な効果が薄い過剰な人格設定
- 毎回固定の長大フォーマット強制
- 根拠のない高得点閾値の自己採点ループ
- あいまい / 競合 / 過剰に強い命令の混在
- モデル仕様が変わりやすい文言の断定（出典や日時なし）

## モデル / バージョン適応

本 Skill は OpenAI 系を主対象とする。ユーザーが「最新」「GPT-5.x」「特定モデル」を指定した場合、日付依存の情報として扱う。

- 公式ドキュメント / 公式Cookbook / 公式ガイドを優先して確認する
- モデル固有の推奨事項と一般原則を分けて記載する
- `GPT-5.4` 指定時は、Prompt guidance・最新モデルガイド・対象ツールガイドを先に見る
- スナップショット指定が必要なら、安定性のため候補も併記する
- 未確認の情報は断定しない
- 出典リンクを添える

OpenAI 系でWeb確認を行う場合は、主に以下を確認する。

- Prompt guidance / Prompting overview
- Prompt optimizer
- 最新モデルガイド / 対象モデルページ（例: GPT-5.4）
- Tools / Tool search / Computer use
- 評価系ガイドや linked evals の説明

## Webリサーチの使い分け

- 日付依存の質問（最新モデル、最新推奨設定、API挙動）は Web で確認する
- 安定した一般論（明確な出力仕様、制約整理、例示の有効性）はローカル知識で対応してよい
- 参照した内容は要約して反映し、長文引用は避ける

  - `references/openai-prompting-best-practices-2026-03.md`

必要な部分だけ読む。全文を毎回読み込まない。

## 境界条件

- 実装コードの作成依頼に変わった場合は、評価・設計支援と実装作業を分離して扱う
- 外部知識が必要な評価は、未確認のまま推測補完しない
- 機密情報が含まれる場合は、貼り付け範囲の縮小や匿名化を提案する
